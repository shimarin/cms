# ADR-007: /api/inquiry エンドポイントの設計

## 状況

問い合わせフォームなどに使えるAPIエンドポイントが必要。

## 決定

### 名前空間

`/api/` 以下を静的ファイル配信から完全に分離したAPIエンドポイント専用とする。
設定ファイルは `docs/api/settings.json` に置く（セオリー上は外側に置くべきだが
わかりやすさを優先）。外部HTTPアクセスは `/api/` ブロックにより不可能。

### XSRFトークン

- GET → トークン発行、サーバーのメモリ上辞書で管理（TTL 1時間）
- POST → `X-XSRF-Token` ヘッダで検証、使い捨て
- 再起動をまたいだアクセスはトークン無効になるが、リトライで対応

セッション管理の複雑さを避けるためメモリ上のシンプルな辞書を採用。

### メール本文テンプレート

テンプレートの先頭行が `Subject:` で始まる場合に件名として抽出する。
メールヘッダとして確立された慣習であり、テンプレートと設定の二重管理を避けられる。

### ハニーポット

`inquiry.honeypot` で指定したフィールドに値が入っていれば、ボットとみなす。
ボット検出時の動作は `inquiry.honeypot_action` で制御する。

- `drop` (デフォルト): メール送信せずに `{"ok": true}` を返す（ボットに気づかせない）。
- `archive`: `Importance: low` および `X-Priority: 5` ヘッダを付与し、さらに件名に接頭辞（デフォルト: `[Bot]`）を付けてメール送信する。Gmailのフィルタによる自動アーカイブ等に利用できる。

接頭辞は `inquiry.honeypot_subject_prefix` でカスタマイズ可能。

レートリミットより実装が単純で、フォームの構造を知らないボットに有効。

### メールアドレスブラックリスト

ハニーポットを突破する既知の送信元を `inquiry.email_blacklist` でブロックできる。
POSTデータの全フィールドを走査し、`@` を含む文字列値をメールアドレス候補として各エントリと照合する（大文字小文字無視）。

- フルアドレス形式: 完全一致
- `@` で始まる形式（例: `"@example.com"`）: `endswith` でドメイン一致

検出時の動作は `honeypot_action` と共通。フィールド名を問わず全フィールドに適用するのは、APIの仕様上メールアドレスのフィールド名が固定されていないため。

### Reply-Toについて

フォーム送信者のアドレスをReply-Toに設定する案があったが見送り。
SMTPのFromヘッダとReply-Toを混同する懸念を排除し、シンプルに保つ。
（Reply-To自体はSPFに影響しないが、混乱を避けるため）
