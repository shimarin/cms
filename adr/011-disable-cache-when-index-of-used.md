# ADR-011: index_of を呼んだページはキャッシュ対象外にする

## 状況

ページの `Last-Modified` / `ETag` には対象ドキュメント自身と採用テンプレートの `mtime` の最大値を採用していた。
しかし `index_of` でディレクトリインデックスを描画するページは、対象ドキュメント自身が更新されていなくても
配下に新しい記事が追加・更新されると表示内容が変わる。この場合、従来の mtime ベースでは
更新が読者のブラウザに伝わらない。

## 選択肢

**A) `index_of` 呼び出し時に走査対象ディレクトリの `max(mtime)` を内部で集約し、
Last-Modified 候補に加える**

正確だが、304 応答はテンプレートをレンダリングしないと判断できなくなり、レンダリング後の
304 判定への移行が必要。テンプレートを介した `include`/`extends` まで考えると静的解析での
事前検出は破綻しやすい。

**B) `index_of` を一度でも呼んだページは丸ごとキャッシュ対象外にする**

実装が極めて単純（フラグを立てて分岐するだけ）。サイトの作りによってはキャッシュ対象から
外れるページが多くなるが、構造上やむを得ない。

## 決定

**B** を採用。

- `render_md_file` の中で `env.globals["index_of"]` を「呼ばれたらフラグを立てる」
  ラッパに差し替える
- 304 判定とキャッシュヘッダ付与をレンダリング後に移動
- フラグが立っていれば `Cache-Control: no-store` を返し、`Last-Modified` / `ETag` は付けず
  304 経路もスキップ
- 立っていなければ従来通り mtime ベースで `Last-Modified` / `ETag` を付け、
  `If-None-Match` / `If-Modified-Since` で 304 応答する

## 制作運用への含意

`index_of` を多用するとキャッシュが効かなくなるため、コンテンツ制作側ではできる限り
`index_of` に頼らず、ドキュメント本文に関連リンクを直接書き入れる方向性を取る。
人手では負担が大きいが、AIを使った制作支援が前提なら現実的。
